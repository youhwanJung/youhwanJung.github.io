원인 분석

ViewModel에 ErrorState? 타입의 필드를 단일 에러 창구로 정의했습니다. 모든 API 호출에서 에러가 발생하면 이 필드에만 값을 할당하고 notifyListeners()를 호출하도록 통일했습니다. ErrorState는 title, message, errorCode를 가진 단순한 데이터 클래스로 정의해, 어떤 에러든 동일한 구조로 표현되도록 했습니다.

UI 전달은 View에서 ChangeNotifier의 변경을 감지하는 리스너가 담당했습니다. 변경이 감지되면 에러 상태가 null인지 확인하고, 값이 있으면 바텀시트로 에러를 표시한 뒤 즉시 에러 상태를 초기화했습니다. 덕분에 에러가 ViewModel → View 단방향으로만 흐르는 구조가 만들어졌습니다.

결과 & 배운 점

에러 처리 코드가 ViewModel 한 곳으로 집약되면서, 새 기능을 추가할 때 에러 필드에 값을 할당하는 패턴만 따르면 됐습니다. 기존에 기능마다 따로 작성하던 에러 관련 상태와 처리 코드가 대폭 줄었고, 에러가 화면에 전달되는 경로도 리스너 하나로 통일되어 어디서 에러가 터졌든 동일한 방식으로 사용자에게 표시됐습니다.

에러도 결국 "상태"라는 것을 체감했습니다. 에러 처리 방식을 통일하는 것보다, 에러가 흐르는 경로 자체를 단일화하는 것이 훨씬 근본적인 해결책이었습니다. ViewModel이 에러 상태를 소유하고, View는 그걸 구독만 하는 구조가 Clean Architecture에서 말하는 단방향 데이터 흐름과도 자연스럽게 맞아떨어졌습니다. 결과적으로 코드량이 줄었을 뿐만 아니라, 나중에 에러 처리 방식을 바꿔야 할 때도 한 곳만 수정하면 되는 구조가 됐습니다.

원인 분석

계좌 연동 없이 송금을 연결하려면 이미 사용자들이 신뢰하고 사용 중인 외부 송금 서비스를 활용하는 수밖에 없었습니다. 문제는 앱에서 외부 앱으로 자연스럽게 연결하는 방법을 몰랐다는 것이었습니다.

결과 & 배운 점

01
API 명세서 없음 -> 개발자도구로 직접 API 명세 추출
문제 상황

이 프로젝트는 회사에서 요청한 것이 아니라 제가 자발적으로 기획하고 개발한 프로젝트였습니다. 기존 출결 시스템은 웹사이트로만 제공되고 있었고, 앱에서 사용할 API 주소나 요청/응답 형식을 정리한 명세서가 전혀 없었습니다. 담당 부서에 API 명세를 요청할 수 있는 상황도 아니었기 때문에, 앱에서 어떤 데이터를 어떻게 받아야 하는지부터 막막했습니다.

원인 분석

API를 새로 만들거나 변경할 수 있는 권한이 없었고, 서버 담당자에게 명세 작업을 따로 요청하기도 어려운 개인 프로젝트였습니다. 결국 기존 API를 그대로 사용하면서, 제가 직접 각 기능이 어떤 요청을 보내고 어떤 응답을 받는지 알아내야 하는 상황이었습니다.

해결 방법

웹사이트의 개발자도구 Network 탭을 열어, 출근/퇴근, 휴가 신청, 근무지 변경 등 모든 기능을 하나씩 직접 사용해보며 각 기능에서 발생하는 요청 URL, 메서드, 파라미터, 응답 데이터를 일일이 기록했습니다. 이렇게 모은 정보를 바탕으로 API 명세를 직접 정리하고, 응답 구조를 그대로 받는 DTO와 앱에서 사용할 도메인 모델을 분리한 뒤, Mapper를 통해 웹의 응답값을 제가 원하는 형태로 변환하는 구조로 설계했습니다.

결과 & 배운 점

API 명세서가 전혀 없는 상태에서도 기존 서비스를 분석해 필요한 데이터를 모두 추출하고, 앱에서 정상적으로 사용할 수 있는 구조까지 만들 수 있었습니다.

명세서가 없다고 해서 개발이 불가능한 것은 아니라는 걸 직접 경험했습니다. 이미 동작하는 서비스가 있다면, 개발자도구만으로도 충분히 API 구조를 파악할 수 있다는 것을 배웠고, DTO와 도메인 모델을 분리하고 Mapper로 변환하는 구조를 미리 잡아두면 서버 응답 형식에 변화가 생기더라도 Mapper 한 곳만 수정해서 대응할 수 있다는 것도 체감했습니다.

02
커스텀 애니메이션의 렌더링 이슈 -> Lottie 도입으로 해결
문제 상황

프로젝트를 진행할 당시 애니메이션에 대해 공부하고 있었고, 출결 앱의 여러 화면 전환과 인터랙션을 직접 애니메이션으로 구현해보고 싶었습니다. 하지만 모든 요소를 AnimationController와 커스텀 애니메이션으로 직접 만들다 보니, 화면에 따라 렌더링 이슈가 발생하거나 애니메이션이 버벅거리는 현상이 생겼습니다.

원인 분석

복잡한 모션이나 디테일한 효과를 직접 구현하려면 다수의 위젯과 컨트롤러를 정교하게 제어해야 했고, 그 과정에서 불필요한 리빌드와 연산이 늘어나면서 성능 저하로 이어졌습니다. 모든 애니메이션을 직접 만드는 방식은 시간 대비 효율도 떨어지고, 완성도 면에서도 한계가 있었습니다.

해결 방법

이런 고민을 하던 중 Lottie를 알게 됐습니다. Lottie는 디자인 툴에서 만든 애니메이션을 JSON 형태로 제공해, 별도의 복잡한 코드 없이 가져와서 재생할 수 있는 라이브러리였습니다. 출근/퇴근 성공, 날씨 표현 등 시각적 효과가 중요한 부분은 Lottie 애니메이션으로 대체하고, 화면 전환처럼 간단한 인터랙션만 커스텀 애니메이션으로 남겨 역할을 나눠 적용했습니다.

결과 & 배운 점

렌더링 이슈와 버벅임 없이 풍부한 애니메이션 효과를 적용할 수 있었고, 개발 시간도 크게 줄었습니다.

모든 것을 직접 구현하는 것이 항상 최선은 아니라는 것을 배웠습니다. 주어진 환경과 상황에 맞게 적절한 도구를 선택해서 사용하는 것도 개발자에게 필요한 역량이라는 걸 체감했고, 효과와 효율 사이에서 균형을 맞추는 경험을 할 수 있었습니다.

03
평범한 연차 버튼 -> 글레어(Glare) 애니메이션으로 시각적 강조
문제 상황

연차 현황 카드에는 '휴가 사용하기' 버튼이 있는데, 직장인이라면 누구나 연차 사용이 반가운 기능이라는 점에서 이 버튼이 화면에서 그냥 묻히는 게 아쉬웠습니다. 단순한 색상 버튼만으로는 시선을 끌기에 부족했고, 다른 정적인 UI 요소들과 구분되는 특별한 느낌을 주고 싶었습니다.

원인 분석

버튼의 의미(연차 사용)와 시각적 표현이 다른 버튼들과 동일하다 보니, 사용자가 굳이 눌러봐야겠다는 동기를 주지 못했습니다. 정적인 디자인 변경만으로는 한계가 있었고, 자연스럽게 시선이 가는 모션 효과가 필요하다는 결론에 도달했습니다.

해결 방법

빛이 버튼 표면을 스치고 지나가는 듯한 글레어(Glare) 애니메이션을 직접 구현했습니다. AnimationController와 CurvedAnimation으로 애니메이션 진행도를 관리하고, LayoutBuilder로 버튼의 실제 너비를 받아와 빛이 이동할 거리를 동적으로 계산했습니다. 빛 막대를 비스듬히 기울인 상태에서도 화면 밖까지 자연스럽게 빠져나가도록 sin 함수로 추가 이동 거리를 보정했고, Stack으로 정적인 버튼과 애니메이션 레이어를 분리해 AnimatedBuilder가 빛 효과 부분만 리빌드하도록 구성했습니다. 애니메이션이 끝나면 addStatusListener와 짧은 딜레이로 다시 시작되게 만들어 일정한 리듬으로 반복되도록 했습니다.

결과 & 배운 점

'휴가 사용하기' 버튼에 은은한 빛 반사 효과가 더해지면서, 화면에서 자연스럽게 눈에 띄는 요소가 되었습니다.

단순한 애니메이션 하나도 위젯 크기, 회전 각도, 리빌드 범위까지 고려해야 자연스럽고 가벼운 효과가 된다는 것을 배웠습니다. 특히 정적인 부분과 애니메이션 부분을 분리해서 설계하는 습관은 이후 다른 화면의 애니메이션을 구현할 때도 성능을 고려하는 기준이 되었습니다.

04
작은 화면에 많은 일정 정보 -> 오버스크롤 기반 확장형 일정 뷰로 해결
문제 상황

일정 화면에는 캘린더와 함께 일주일 단위의 일정 리스트를 보여줘야 했습니다. 하지만 모바일이라는 한정된 화면에서 캘린더와 일정 리스트, 그리고 사용자가 더 보고 싶어할 수 있는 추가 일정까지 모두 한 화면에 담기에는 공간이 턱없이 부족했습니다. 모든 정보를 한 번에 펼쳐두면 캘린더가 차지하는 영역이 커서 리스트가 잘려 보이거나, 정보량 자체가 너무 많아 가독성이 떨어지는 문제가 있었습니다.

원인 분석

평소에는 일주일 정도의 일정만 보여주는 것이 가독성 면에서 가장 적절했지만, 사용자가 더 많은 일정을 확인하고 싶을 때를 위한 동작이 없었습니다. 화면을 전환해 별도의 전체 일정 페이지로 이동하는 방식은 흐름이 끊기고, 처음부터 모든 일정을 무한 리스트로 보여주는 방식은 캘린더 영역을 압박해 오히려 가독성을 해쳤습니다. 결국 평소와 '더 보기' 상태를 자연스럽게 전환할 수 있는 인터랙션이 필요했습니다.

해결 방법

일정 리스트를 아래로 오버스크롤하면, NotificationListener로 오버스크롤 정도를 감지해 화면 상단에 안내 화살표를 Lottie 애니메이션으로 보여주도록 했습니다. 일정 임계값을 넘으면 트리거가 활성화되고, AnimatedPositioned로 캘린더 영역이 위로 축소되면서 일정 리스트 영역이 화면 대부분을 차지하도록 자연스럽게 확장됐습니다. 확장된 상태에서는 기존 일주일치 데이터 대신 무한 스크롤 리스트로 전환되어, 사용자가 스크롤할 때마다 위/아래 방향에 따라 추가 일정을 가져오는 API를 호출하도록 구현했습니다. 닫기 버튼을 누르면 다시 원래의 캘린더 + 일주일 일정 뷰로 복귀했습니다.

결과 & 배운 점

사용자는 평소엔 캘린더와 간결한 일주일 일정을 보고, 더 많은 정보가 필요할 때만 오버스크롤이라는 자연스러운 동작으로 화면을 확장해 무한히 일정을 탐색할 수 있게 됐습니다. 화면 전환 없이도 정보 밀도를 상황에 맞게 조절할 수 있었습니다.

문제 상황

개발 중에는 제 테스트 기기에서 UI가 의도한 대로 잘 보였는데, TestFlight와 APK로 실제 동료들에게 배포하자 화면 곳곳에서 문제가 보고되기 시작했습니다. 일부 기기에서는 텍스트가 잘려 보이거나, 버튼과 카드 레이아웃이 깨지거나, 글자 크기가 화면에 비해 너무 크거나 작게 보이는 경우가 있었습니다.

원인 분석

제가 가진 테스트 기기는 한정적이었지만, 동료들이 사용하는 아이폰과 안드로이드 기종은 화면 비율과 해상도, 시스템 글꼴 크기 설정까지 매우 다양했습니다. 개발 단계에서 고정된 크기 값으로 작성된 위젯들은 특정 화면 비율에서는 의도한 대로 보였지만, 화면이 더 좁거나 길쭉한 기기에서는 레이아웃이 그대로 무너지는 구조였습니다.

문제 상황

앱은 화면 하단에 떠 있는(absolute, 둥근 모서리) 커스텀 바텀 탭 바를 사용합니다. 그런데 홈 탭에서 학적정보, 알람 확인, 게시글 상세 같은 하위 화면으로 들어가면, 그 화면들 위에도 하단 탭 바가 그대로 떠 있어서 본문 내용과 버튼을 가리는 문제가 있었습니다. 탭 바는 첫 화면(목록)에서만 보여야 하는데, 같은 스택 안에서 push된 상세 화면에서도 계속 노출되는 상황이었습니다.

원인 분석

React Navigation에서 바텀 탭 바는 "탭에 연결된 스택" 전체에 적용됩니다. 즉 홈 탭 안의 Stack Navigator에 상세 화면들을 같이 등록해두면, 그 스택 안에서 아무리 화면을 push해도 부모인 탭 네비게이터 입장에서는 여전히 "홈 탭"에 머물러 있는 것이라 탭 바가 사라지지 않았습니다. 탭 바의 노출 여부가 "현재 탭"이 아니라 "현재 스택 안에서 실제로 어떤 화면을 보고 있는지"에 따라 결정되어야 한다는 것이 핵심이었습니다.

해결 방법

두 가지 전략을 함께 적용했습니다. 첫째, 각 스택 네비게이터에서 getFocusedRouteNameFromRoute로 현재 포커스된 화면 이름을 읽어와, useLayoutEffect 안에서 첫 화면(예: MainScreen)일 때만 탭 바 스타일을 정상 노출하고, 그 외 하위 화면에서는 tabBarStyle: { display: 'none' }로 숨기도록 동적으로 제어했습니다. 둘째, 게시물 작성/수정/상세처럼 탭 바가 절대 보이면 안 되는 화면은 탭 스택이 아니라 최상위 RootStack으로 분리해서, 구조적으로 탭 바 영향권 밖에 두었습니다.

목록 화면에서는 하단 탭 바가 떠 있고, 상세 화면으로 들어가면 탭 바가 자연스럽게 사라져 콘텐츠를 가리지 않는 의도한 UX를 완성했습니다.

탭 바의 노출 여부는 "어떤 탭인지"가 아니라 "스택 안에서 실제로 보고 있는 화면이 무엇인지"로 판단해야 한다는 것을 배웠습니다. 또한 모든 화면을 탭 스택에 욱여넣기보다, 탭 바가 필요 없는 화면은 RootStack으로 분리하는 식으로 네비게이터 계층을 설계하면 UI 제어가 훨씬 단순해진다는 것도 체감했습니다.

메인 화면 진입 때마다 로딩이 느림 -> 독립 API 병렬 호출로 응답 시간 단축
문제 상황

메인 화면은 프로필, 포인트, 알림 수, 학교/학과 공지사항, 인기글, 공모전, 이벤트 등 한 화면에서 보여줄 데이터가 매우 많았습니다. 그런데 화면에 처음 들어오거나 상세 화면에서 뒤로 돌아올 때마다 데이터가 모두 채워지기까지 눈에 띄게 시간이 걸렸고, 화면이 비어 있다가 한참 뒤에 한 번에 그려지는 느낌이라 사용자 입장에서 앱이 느리고 답답하게 느껴졌습니다.

결과 & 배운 점

여러 데이터를 한 화면에 모아 보여주면서도, 메인 화면 진입 시 체감 로딩 시간을 크게 줄여 데이터가 거의 동시에 채워지도록 개선할 수 있었습니다.

비동기 처리에서 await를 순서대로 늘어놓으면 "병렬로 처리할 수 있는 일까지 직렬로 기다리게 된다"는 것을 직접 체감했습니다. 요청 간에 의존 관계가 있는지 먼저 따져보고, 독립적인 요청은 Promise.all로 묶는 습관이 성능에 큰 차이를 만든다는 것을 배웠습니다.

서버 지연 시 요청이 무한 대기 -> AbortController로 타임아웃을 직접 구현
문제 상황

친구코드 이벤트에서는 코드를 등록하면 친구에게 알림 전송, 포인트 적립, 포인트 이력 기록 같은 여러 부가 요청이 연달아 실행됩니다. 그런데 특정 서버 요청의 응답이 느리거나 오지 않을 때, 앱이 그 요청을 무한정 기다리며 다음 단계로 넘어가지 못하고 멈춘 것처럼 보이는 문제가 있었습니다. 사용자는 등록이 됐는지 안 됐는지 알 수 없는 상태로 방치됐습니다.

원인 분석

기본 fetch에는 타임아웃 개념이 없어, 서버가 응답을 늦게 주거나 응답하지 않으면 요청이 끝나지 않고 계속 대기 상태에 머무릅니다. 알림 전송처럼 "꼭 성공하지 않아도 되는" 보조 요청까지 메인 흐름을 붙잡고 있다 보니, 한 요청의 지연이 전체 등록 과정을 멈추게 만드는 구조였습니다.

해결 방법

AbortController와 setTimeout을 함께 사용해 요청별로 타임아웃을 직접 구현했습니다. 요청을 보낼 때 일정 시간(중요 요청 5초, 보조 알림 요청 2초) 뒤에 controller.abort()가 호출되도록 예약하고, fetch에 signal로 연결했습니다. 요청이 정상 완료되면 clearTimeout으로 예약을 해제하고, 시간 초과로 중단된 경우 발생하는 AbortError는 흐름을 깨지 않도록 별도로 구분해 처리했습니다. 이로써 느린 보조 요청이 전체 등록 과정을 막지 않게 했습니다.

결과 & 배운 점

서버 응답이 늦어져도 정해진 시간이 지나면 요청을 스스로 중단하고 다음 단계로 넘어가도록 만들어, 앱이 멈춘 것처럼 보이는 현상을 없앴습니다.

fetch에는 타임아웃이 없다는 점, 그리고 네트워크는 항상 "느리거나 실패할 수 있다"는 전제로 설계해야 한다는 것을 배웠습니다. 특히 부가적인 요청은 실패하거나 중단되더라도 핵심 기능 흐름을 막지 않도록 분리해 두는 것이 안정적인 UX로 이어진다는 것을 체감했습니다.

일정 시간 후 모든 DB 요청이 멈춤 -> 커넥션 풀 누수 차단으로 안정화
문제 상황

앱을 어느 정도 사용하다 보면, 멀쩡히 동작하던 서버가 갑자기 모든 데이터 요청에 응답하지 않고 멈추는 현상이 발생했습니다. 게시글 목록, 출석, 포인트 등 어떤 기능을 호출해도 응답이 돌아오지 않았고, 서버를 재시작하면 잠시 정상으로 돌아왔다가 시간이 지나면 다시 같은 증상이 반복됐습니다.

원인 분석

서버는 매 요청마다 DB 커넥션 풀(connectionLimit: 10)에서 커넥션을 하나씩 빌려 쿼리를 실행하는 구조였습니다. 그런데 쿼리 도중 에러가 나거나 특정 경로에서 커넥션을 반환(release)하지 않으면, 그 커넥션이 풀로 돌아오지 않고 계속 점유된 상태로 남았습니다. 이렇게 반환되지 않은 커넥션이 쌓여 결국 10개를 모두 소진하면, 이후의 모든 요청은 빈 커넥션이 생기기만 무한정 기다리며 멈춰버리는 것이 원인이었습니다.

해결 방법

모든 DB 접근 함수를 try / catch / finally 구조로 통일하고, 성공하든 예외가 발생하든 항상 finally 블록에서 if (conn) conn.release()로 커넥션을 풀에 되돌려주도록 규칙을 강제했습니다. 즉 "커넥션을 빌리면 어떤 경로로 끝나도 반드시 반환한다"를 모든 쿼리 함수에 동일하게 적용해 누수 지점을 원천적으로 없앴습니다. 더불어 모든 쿼리는 문자열을 직접 이어 붙이지 않고 ? 파라미터 바인딩으로 값을 전달하도록 해, SQL 인젝션 위험까지 함께 차단했습니다.

결과 & 배운 점

커넥션이 항상 제때 반환되도록 만든 뒤로는, 장시간 사용해도 서버가 멈추는 현상 없이 안정적으로 요청을 처리할 수 있게 되었습니다.

한정된 자원(커넥션 풀)은 "빌리는 코드"보다 "돌려주는 코드"가 더 중요하다는 것을 배웠습니다. 특히 예외 상황에서도 자원이 반드시 해제되도록 finally로 보장하는 패턴을 일관되게 적용하는 것이, 당장 눈에 보이지 않지만 운영 중에 치명적인 장애를 막는 핵심이라는 것을 체감했습니다.
