01
에러 처리 파편화 -> _errorState 도입으로 단일 창구 구축
문제 상황

기능이 늘어날수록 try-catch가 여기저기 생겨나기 시작했다. API 호출, 소셜 로그인, 파일 처리 등 기능마다 에러를 따로 잡고 따로 처리하다 보니 에러 처리 방식이 제각각이 됐고, ViewModel에서 발생한 에러를 화면에 어떻게 전달해야 할지 기준조차 없었다.

원인 분석

원인은 에러 처리를 책임지는 단일 창구가 없는 구조였기 때문이다. 기능마다 각자 에러를 잡다 보니 관련 상태와 코드가 자연스럽게 흩어질 수밖에 없었고, "어떻게 하면 에러를 한 곳에서 일관성 있게 처리하고 화면에 넘겨줄 수 있을까?"라는 고민으로 이어졌다.

해결 방법

ViewModel에 ErrorState? 타입의 필드를 단일 에러 창구로 정의했다. 모든 API 호출에서 에러가 발생하면 이 필드에만 값을 할당하고 notifyListeners()를 호출하도록 통일했다. ErrorState는 title, message, errorCode를 가진 단순한 데이터 클래스로 정의해, 어떤 에러든 동일한 구조로 표현되도록 했다.

UI 전달은 View에서 ChangeNotifier의 변경을 감지하는 리스너가 담당했다. 변경이 감지되면 에러 상태가 null인지 확인하고, 값이 있으면 바텀시트로 에러를 표시한 뒤 즉시 에러 상태를 초기화했다. 덕분에 에러가 ViewModel → View 단방향으로만 흐르는 구조가 만들어졌다.

결과 & 배운 점

에러 처리 코드가 ViewModel 한 곳으로 집약되면서, 새 기능을 추가할 때 에러 필드에 값을 할당하는 패턴만 따르면 됐다. 기존에 기능마다 따로 작성하던 에러 관련 상태와 처리 코드가 대폭 줄었고, 에러가 화면에 전달되는 경로도 리스너 하나로 통일되어 어디서 에러가 터졌든 동일한 방식으로 사용자에게 표시됐다.

에러도 결국 "상태"라는 것을 체감했다. 에러 처리 방식을 통일하는 것보다, 에러가 흐르는 경로 자체를 단일화하는 것이 훨씬 근본적인 해결책이었다. ViewModel이 에러 상태를 소유하고, View는 그걸 구독만 하는 구조가 Clean Architecture에서 말하는 단방향 데이터 흐름과도 자연스럽게 맞아떨어졌다. 결과적으로 코드량이 줄었을 뿐만 아니라, 나중에 에러 처리 방식을 바꿔야 할 때도 한 곳만 수정하면 되는 구조가 됐다.

02
영수증 OCR 처리 시간 6~10초 -> ML Kit 1차 추출로 1~2초 단축
문제 상황

영수증 이미지를 서버로 전송하면 AI가 직접 OCR을 수행하고 카테고리까지 분류하는 방식을 사용했는데, 사용자가 영수증을 등록하고 결과를 받기까지 6~10초가 걸렸다. 서버 응답을 기다리는 동안 사용자 입장에서는 그냥 멈춰있는 화면을 보고 있어야 했고, 체감 대기 시간이 너무 길었다.

원인 분석

이미지 자체를 서버로 넘기면 서버가 이미지 디코딩 → OCR → 텍스트 파싱 → AI 카테고리 분류까지 모든 작업을 순차적으로 처리해야 했기 때문이다. 이미지 용량이 클수록 전송 시간까지 붙어 전체 처리 시간이 길어질 수밖에 없는 구조였다.

해결 방법

ML Kit를 클라이언트에 도입해 텍스트 추출을 기기에서 먼저 처리하도록 역할을 분리했다. ML Kit는 실무에서 QR코드 인식 기능을 구현할 때 접한 적이 있었는데, 그때 공부하면서 OCR 기능도 함께 제공한다는 것을 알고 있었고 이번에 자연스럽게 떠올렸다. TextRecognizer로 영수증 이미지에서 텍스트 블럭을 1차 추출하고, 서버가 받을 수 있는 구조로 정규화한 뒤 전송했다. 덕분에 서버는 이미지 대신 텍스트 데이터만 받아 AI 카테고리 분류에만 집중할 수 있게 됐다.

결과 & 배운 점

6~10초 걸리던 처리 시간이 1~2초로 크게 단축됐다. 클라이언트와 서버의 역할을 명확히 나눈 것만으로 사용자 체감 속도가 눈에 띄게 개선됐다.

무거운 작업을 서버에 몰아주는 것보다, 기기에서 할 수 있는 건 기기에서 먼저 처리하고 서버엔 꼭 필요한 데이터만 넘기는 방식이 성능 면에서 훨씬 효율적이라는 걸 체감했다. 또한 실무나 사이드 프로젝트에서 접했던 기술이라도 깊게 파고들어 두면, 예상치 못한 순간에 문제 해결의 실마리가 된다는 것도 배웠다.

03
계좌 연동 없이 송금 플로우 구현 — 앱 스킴(App Scheme) 도입
문제 상황

정산 기능을 구현할 때 사용자가 정산 알림을 받고 바로 송금까지 이어지는 플로우가 필요했다. 실제 계좌 연동이 가장 자연스러운 방법이었지만, 금융 API 연동은 인증 절차와 보안 요건이 까다로워 현재 개발 단계에서 바로 적용하기 어려웠다. 송금 플로우 없이는 정산 기능이 반쪽짜리가 될 수밖에 없었다.

원인 분석

계좌 연동 없이 송금을 연결하려면 이미 사용자들이 신뢰하고 사용 중인 외부 송금 서비스를 활용하는 수밖에 없었다. 문제는 앱에서 외부 앱으로 자연스럽게 연결하는 방법을 몰랐다는 것이었다.

해결 방법

앱 스킴(App Scheme)이라는 개념을 알게 됐다. 앱 스킴은 kakaopay://, supertoss:// 같은 URI 형태로 특정 앱을 직접 실행할 수 있는 방식이다. canLaunchUrl()로 해당 앱이 기기에 설치되어 있는지 먼저 확인하고, 설치되어 있으면 앱을 직접 실행하고 없으면 웹 페이지로 fallback하도록 구현했다. 현재는 토스와 카카오페이 두 가지를 지원하고 있다.

결과 & 배운 점

계좌 연동 없이도 정산 알림에서 송금 앱으로 바로 이어지는 플로우를 구현할 수 있었다. 앱에서 앱으로 이동하는 딥링크 개념을 처음 접했고, 모바일 생태계에서 앱 간 연결이 어떻게 이루어지는지 배웠다.

완벽한 계좌 연동은 아니지만 현실적인 제약 안에서 사용자 경험을 최대한 끊기지 않게 만드는 방법을 고민한 경험이었다. 기술적으로 막히는 부분이 있을 때 완전한 해결책만 고집하기보다, 현재 상황에서 최선의 대안을 찾는 것도 개발의 중요한 부분이라는 걸 배웠다.

04
소셜 로그인과 로컬 회원가입 간 이메일 중복 — 가입 경로 충돌 처리
문제 상황

Google, Kakao, Naver, Apple, 로컬까지 5가지 로그인 방식을 지원하다 보니 예상치 못한 문제가 생겼다. Naver 계정 이메일과 동일한 이메일로 로컬 회원가입을 했을 때 서버에서 회원 정보가 겹치는 상황이 발생했다. 같은 이메일이지만 가입 경로가 다른 두 사용자가 동일인으로 처리되거나 충돌이 생기는 케이스였다.

원인 분석

소셜 로그인은 이메일을 식별자로 사용하는데, 로컬 회원가입도 이메일 인증 방식이다 보니 동일한 이메일이 서로 다른 가입 경로에서 중복으로 존재할 수 있는 구조였다. 소셜 로그인을 처음 구현하면서 가입 경로가 다를 때 이메일이 겹치는 케이스를 미처 고려하지 못했던 것이 원인이었다.

해결 방법

서버와 협의해 로그인 시 해당 이메일로 가입된 계정이 있는지, 있다면 어떤 경로로 가입했는지를 응답에 함께 내려주도록 했다. 클라이언트에서는 서버 응답을 기반으로 신규 유저인지 기존 유저인지, 다른 소셜 로그인으로 이미 가입된 계정인지를 판단해 분기 처리했다. 이미 다른 경로로 가입된 이메일이면 "이미 다른 소셜 로그인으로 가입된 계정입니다"라는 안내 팝업을 띄워 사용자가 혼란 없이 올바른 로그인 방식을 선택할 수 있도록 했다.

결과 & 배운 점

이메일 중복으로 인한 회원 정보 충돌 문제를 해결했다. 소셜 로그인을 처음 구현하면서 단순히 각 플랫폼 SDK 연동에만 집중했는데, 여러 가입 경로가 공존할 때 동일한 이메일이 충돌할 수 있다는 사실을 직접 겪으면서 알게 됐다.

인증 시스템을 설계할 때 이메일만으로 사용자를 식별하는 것이 아니라, 가입 경로(provider)를 식별자의 일부로 함께 관리해야 한다는 것을 배웠다. 혼자 해결하려 하지 않고 서버 개발자와 빠르게 협의해 응답 구조를 함께 설계한 것이 문제를 깔끔하게 해결하는 데 큰 도움이 됐다.

05
FCM 개발 · 운영 환경 토큰 불일치 — APNs 환경 분리 이해
문제 상황

FCM 알림을 iOS에 적용하면서 APNs 인증서를 개발(Sandbox)과 운영(Production) 두 가지를 Firebase에 등록했다. 개발 환경에서 직접 FCM 토큰을 뽑아 서버에 전달하고 테스트했을 때는 알림이 정상적으로 왔다. 그런데 TestFlight로 운영 환경에서 똑같은 방식으로 테스트하니 알림이 전혀 오지 않았다.

원인 분석

개발 환경과 운영 환경에서 FCM이 기기에 발급하는 토큰이 서로 다르다는 것을 몰랐던 게 원인이었다. APNs 자체가 Sandbox와 Production 환경을 완전히 분리해서 운영하기 때문에, 같은 기기라도 개발 빌드와 운영 빌드에서 발급받는 FCM 토큰이 달랐다. 개발 환경에서 뽑은 토큰을 운영 환경에서 그대로 사용했으니 당연히 전달이 안 됐던 것이었다.

해결 방법

TestFlight 빌드에서 FCM 토큰을 새로 뽑아 하드코딩으로 전달하는 방식 대신, 로그인 시마다 FCM 토큰을 서버에 PATCH 요청으로 갱신하도록 구현했다. 이렇게 하면 개발 환경이든 운영 환경이든 로그인할 때마다 해당 환경에 맞는 토큰이 자동으로 서버에 등록되기 때문에, 환경을 바꿔가며 테스트할 때도 토큰 불일치 문제가 생기지 않았다.

결과 & 배운 점

로그인 시마다 환경에 맞는 FCM 토큰이 자동으로 서버에 등록되면서 개발과 운영 환경을 오가며 테스트할 때도 토큰 불일치 문제가 사라졌다.

FCM이 단순히 메시지를 전달하는 도구가 아니라, iOS의 경우 APNs 환경에 따라 토큰 자체가 달라진다는 것을 알게 됐다. 개발 환경에서 잘 됐다고 운영 환경에서도 당연히 될 거라고 생각했는데, 플랫폼마다 환경 분리 방식이 다르다는 것을 직접 겪으면서 배웠다.
